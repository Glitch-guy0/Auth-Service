## Technical Requirements

### TypeORM Entity Definition
- Use `@Entity()` decorator to define the table name (`public_key_registry`)
- Use `@PrimaryColumn('uuid')` for kid (UUID v7, not auto-generated)
- Use `@Column({ type: 'text' })` for public_key
- Use `@CreateDateColumn()` for created_at
- Use `@Column({ type: 'timestamp' })` for expires_at

### Field Specifications
- **kid**: UUID v7, primary key, unique identifier for the key
- **public_key**: text, PEM-encoded public key (RSA or EC), not nullable
- **created_at**: timestamp, auto-generated on creation
- **expires_at**: timestamp, not nullable (key expiration time for rotation)

### Database Configuration
- Table name: `public_key_registry`
- Use snake_case for column names
- Enable timestamps for audit trail
- Prepare for key rotation (multiple keys may be active during rotation window)

### UUID v7 Considerations
- UUID v7 provides time-ordered identifiers
- Use a UUID v7 library (e.g., `uuidv7` or generate manually)
- kid is NOT auto-generated by TypeORM — must be generated in application code before insert
- Time-ordered UUIDs support efficient database indexing

### Entity Code Structure
```typescript
import { Entity, PrimaryColumn, Column, CreateDateColumn } from 'typeorm';

@Entity('public_key_registry')
export class PublicKeyRegistry {
  @PrimaryColumn('uuid')
  kid: string;

  @Column({ type: 'text' })
  public_key: string;

  @CreateDateColumn()
  created_at: Date;

  @Column({ type: 'timestamp' })
  expires_at: Date;
}
```

### Import Paths
- Import from `typeorm` for decorators
- Use path alias `@modules/key` for cross-module imports

### Design Rationale
- `kid` (Key ID) is the JWT `kid` header value — used to select which public key verifies a given JWT
- UUID v7 provides time-ordered IDs that support efficient range queries for key rotation
- `expires_at` enables key rotation: new keys are inserted before old keys expire, allowing a grace period
- `public_key` as text stores PEM-encoded keys (RSA 2048+ or EC P-256/P-384)

### Key Management
- Each JWT signing key has a unique `kid` (Key ID)
- JWTs include `kid` in the header to indicate which key signed them
- Verifier looks up `kid` in PublicKeyRegistry to find the correct public key
- `expires_at` enables automatic key rotation without service downtime
- Multiple keys may be active simultaneously during rotation windows

## Implementation Notes

### Key Considerations
- Use TypeORM decorators for database mapping
- Keep entity definition simple and focused
- No business logic in entity
- kid is NOT auto-generated — must be generated in application code using UUID v7
- Use a UUID v7 library for generating kid values
- public_key should store PEM-encoded keys (-----BEGIN PUBLIC KEY-----)
- expires_at should use timestamp type for proper date handling

### Common Pitfalls
- Don't use PrimaryGeneratedColumn — kid is generated by the application, not the database
- Ensure UUID v7 is used (time-ordered) rather than UUID v4 (random)
- Use @PrimaryColumn, not @Column with unique constraint
- Store PEM-encoded keys, not raw key material
- Don't forget to handle key rotation logic in the service layer (future story)

### Future Enhancements
- Add index on expires_at for key rotation cleanup queries
- Add index on kid for efficient JWT verification lookups
- Add soft delete capability for revoked keys
- Add key metadata (algorithm, key size) for validation
- Consider adding a `revoked_at` field for emergency key revocation
