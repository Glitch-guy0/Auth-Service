- (insight) User has SQL+NoSQL hybrid: user table SQL, demographics NoSQL, refresh keys SQL
- (insight) Personal project, simple auth now, OAuth consumer + provider future, scalability focus
- (decision) NoSQL for logging (experimental, may not sustain), all tokens bcrypt hashed, user tokens separated into own table
- (decision) Logout: blacklist access token in Redis (NestJS guard) + delete refresh token from DB
- (question) Where does access token live? JWT vs opaque? RSA table still needed? Token expiry windows? 1:1 or 1:N token relationship?
- (insight) JWT payload: userid, role, authenticated_at, server_id, kid. Private key generated on startup, deleted from memory after signing
- (insight) Signup: username/email unique check, bcrypt password, default values. Login: username/email lookup, blocked check, bcrypt validate
- (decision) Token expiry: 1 day access (tunable), 1 week refresh (tunable). Refresh in cookie, access in response
- (question) Two key providers meaning? Key rotation strategy? Cookie security config? Refresh token flow? Refresh token storage in DB?
- (insight) Setup script generates keys.json with kid (uuidv7), public/private keys, metadata. Private key never in DB, only in file
- (question) keys.json protection? Key expiry handling? Private key read frequency? Multi-instance key sharing? Metadata field purposes?
- (decision) Token refresh: rotate on each use, delete old + insert new on each login
- (insight) Environment management: .env → env reader → validator → modules. Each module calls appContext() for its env vars. Main.js entry point
- (question) Vault functionality scope? Validation rules? Module context pattern (DI vs locator)? Dev/prod swapping method? Hot reload capability?
- (decision) keys.json: file permissions. Expired key: fatal error, graceful shutdown. Private key read: every signing. Multi-instance: own keys.json, public from DB. Metadata: audit trail
- (decision) Vault: abstraction layer, hexagonal architecture required. Validation: injected module requirements + type checks. setup() function, appContext global variable. Same .env dev/prod. Restart required for config changes
- (question) Database choice (PostgreSQL?). Redis scope (blacklist + rate limiting?). Framework (NestJS confirmed?). Bcrypt cost factor?
- (decision) Databases: PostgreSQL + MongoDB. Redis: blacklisting only for now. Framework: NestJS. Bcrypt cost: 10
- (insight) Pattern: Not just auth system but auth platform. Hexagonal architecture enables OAuth provider future without rewrite. keys.json per-instance solves multi-instance key distribution
- (insight) Logging: loggerService (debug/warn/info/error/fatal) → logProvider → Telemetry Factory. Metrics and Tracing planned as future (opacity 30)
- (question) Log provider implementation? Telemetry factory scope? Log levels by environment? Log storage destination? Fatal log behavior?
- (decision) Logging: pino + chalk (console). Logs only for now. Levels via .env. Console output only. Fatal: graceful shutdown with module shutdown() methods
- (insight) Module lifecycle: setup() → run() → shutdown() (optional). All major components designed. Ready to converge or continue with RBAC, password reset, rate limiting, API contracts
- (insight) All planning artifacts created: Architecture doc, Technical documentation, README. Ready for implementation.
